Last updated: {{ current_year }}-05-15. Working draft — we're committed to making Tiaki accessible and this page lists what we've done so far, what we know is still missing, and how to tell us about barriers you hit.

Our commitment

Conservation belongs to everyone who shows up to do the work. Tiaki aims to meet Web Content Accessibility Guidelines (WCAG) 2.1 Level AA across every interactive surface. Where we fall short today, it is listed below — not hidden.

What we've built

Keyboard

  • Every primary action is reachable by Tab without a mouse, including the topbar hamburger overflow drawer (ARIA menu pattern with roving tab-index, ESC closes, outside-click closes).
  • Form fields use native HTML controls so screen-reader announcements match the visual control type.
  • Skip-to-content links exist on dashboards with heavy chrome.

Screen readers

  • Hero banners, navigation regions, and the page-head strip carry aria-label values describing their purpose.
  • Icon-only buttons (hamburger, dismiss, social) include aria-label; decorative icons are marked aria-hidden="true" so they don't double up.
  • Form errors are wired with aria-describedby pointing at the error message so the failure reads inline with the field name.
  • Status-bearing chips (Public / Private, Currently applied) pair an icon with text — never colour alone.

Visual

  • The default Tiaki palette is checked against WCAG AA contrast minimums (sidebar text on --pf-800 ≈ 13.7:1).
  • Theme presets (Default Tiaki, Forest, Coastal, Tussock) are tuned to similar contrast targets, but custom themes a Coordinator picks are not automatically validated — see the limitations section.
  • Focus rings are explicit and visible across both light (light strips) and dark (navbar) chrome contexts.

Known limitations

  • Custom theme contrast. Coordinators can pick a heading/body colour pair that drops below WCAG AA contrast. The theme editor doesn't yet warn when the chosen pair fails. Tracked as a future improvement.
  • Maps. Some map-based controls (trap-line plotting on the dashboard map) are keyboard-reachable but not yet fully labelled for screen readers. We are working on alternative tabular views.
  • Charts. Reports charts have title and axis labels but do not yet expose a data-table fallback; coming soon.
  • Mobile. Tap targets meet WCAG 2.5.5 (44×44 px) across primary actions; we are still tightening a small number of secondary controls.

Browser and assistive-technology compatibility

We test against current versions of:

  • Chrome and Edge with built-in screen-reader support.
  • Firefox with NVDA on Windows.
  • Safari with VoiceOver on macOS and iOS.

Internet Explorer is not supported. Browsers older than 18 months may not render the platform's color-mix()-derived palette correctly — chrome will still work, but theme colours may degrade to the hardcoded Tiaki defaults.

Tell us about a barrier

If you hit an accessibility issue on Tiaki — even a small one — please report it. Include the page URL, the assistive technology you were using (if any), and what you were trying to do. Reports go to a platform administrator. A dedicated accessibility contact channel will land alongside the formal launch.
